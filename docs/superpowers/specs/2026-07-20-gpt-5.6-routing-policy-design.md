# GPT-5.6 Routing Policy Design

## Objective

Make the provider's Amplifier routing policy deterministic, documented, and
usable when GPT-5.6 availability is incomplete, without moving orchestration
or account-specific policy into the provider implementation.

## Boundaries

- PR #8 remains the only implementation surface.
- `release/gpt-5.6-provider-recovery` remains the only release branch.
- The provider continues to honor an explicit `request.model` exactly.
- The provider default remains `gpt-5.6-sol`; catalog-aware fallback belongs to
  Amplifier routing, not to provider request construction.
- Tests use synthetic catalog fixtures. They do not collect or record a user's
  subscription plan, account catalog, tokens, or other personal data.
- No route may invent `gpt-5.6-codex`. Coding discovers Codex models through the
  existing catalog glob.

## Policy

The policy separates documented model-family facts from project choices:

- OpenAI documents Sol as the flagship model, Terra as the balanced model, and
  Luna as the efficient high-volume model.
- This project therefore chooses Sol first for general and quality-oriented
  work, Luna first for the `fast` role, and Terra as the first balanced fallback.
- Coding prefers a Codex model actually present in the catalog, then Terra,
  Sol, and finally any available GPT-5 model.
- Reasoning and critique prefer Sol with explicit effort, then Terra with an
  explicit effort, and finally any GPT-5 model without an unsupported effort
  assumption.

Exact candidate order:

| Roles | Candidate order |
| --- | --- |
| `general`, `ui-coding`, `creative`, `writing`, `vision`, `image-gen` | Sol, Terra, Luna, `gpt-5.*` |
| `fast` | Luna, Terra, Sol, `gpt-5.*` |
| `coding` | `gpt-?.?-codex*`, Terra, Sol, `gpt-5.*` |
| `reasoning`, `security-audit`, `research`, `critical-ops` | Sol/high, Terra/high, `gpt-5.*` |
| `critique` | Sol/max, Terra/xhigh, `gpt-5.*` |

## Availability behavior

Routing resolves the first candidate present in the available catalog:

- A full catalog selects the intended GPT-5.6 family member.
- A partial catalog without Sol selects Terra where applicable.
- An emergency catalog containing only a legacy GPT-5 model still resolves
  every role through the final glob.

This makes role-based, unconfigured Amplifier requests resilient without adding
network calls, hidden selection, or autonomous fallback behavior to
`ChatGPTProvider._build_payload()`.

## Verification

A routing contract test loads the shipped YAML and verifies:

1. the exact 13-role set;
2. exact candidate order and reasoning effort;
3. full, partial, and emergency synthetic catalog resolution;
4. absence of `gpt-5.6-codex`; and
5. continued explicit request-model precedence in provider tests.

The README and YAML comments explain which statements are public model facts
and which are this repository's routing policy.
